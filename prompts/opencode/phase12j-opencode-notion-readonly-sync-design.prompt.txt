Purpose:
Design a read-only local Notion sync path for OpenCode. This prompt is for planning and design only. Do not implement unless explicitly approved.

Context:
- Notion database: Bazzite Local Gemma Agent — Phase Tracker
- URL: https://www.notion.so/6af4cb83c91d416a9917a631dcf69449?v=5389736d2d3740c0b3f03d1aa55221b4
- Data source: collection://c2918acb-dddf-40ac-9787-ab1254199ff2
- OpenCode does NOT have Notion connector by default
- ChatGPT has Notion connector and can read/write

Prerequisites (ALL must be true):
- [ ] Human has read docs/phase12/NOTION_OPENCODE_SYNC_POLICY.md

Hard Boundaries:
- Do not store Notion tokens in the repo.
- Do not print Notion secrets, tokens, or auth files.
- Do not create a Notion API token in this phase.
- Do not update Notion directly in this phase.
- Do not modify Notion database schema.
- Do not install packages.
- Do not use sudo.
- Do not modify system/security configs.

Allowed:
- Design read-only sync approaches
- Document pros/cons of each approach
- Create design doc
- Create example code (no secrets)
- Update repo docs

Approaches to Evaluate:

Approach 1: Manual Export (Current)
- User exports Notion as CSV/JSON/Markdown
- OpenCode reads exported file
- Pros: Simple, no token needed
- Cons: Manual step, not real-time

Approach 2: ChatGPT Snapshot Relay
- ChatGPT reads Notion via connector
- ChatGPT generates snapshot
- Snapshot shared with OpenCode (via file or paste)
- OpenCode compares to repo
- Pros: Leverages existing connector, no new token
- Cons: Requires ChatGPT session, not automated

Approach 3: User-Local Notion API Token
- User creates Notion integration (outside repo)
- Token stored in ~/.config/bazzite-security/keys.env
- OpenCode reads token from env var
- OpenCode queries Notion API directly
- Pros: Real-time, automated
- Cons: Requires token management, rate limits

Approach 4: MCP Bridge Extension
- Extend existing OpenCode MCP bridge
- Add Notion read tools
- Token managed by bridge
- OpenCode queries via bridge
- Pros: Unified interface, secure token handling
- Cons: Requires bridge code changes

Design Doc Requirements:
- Compare all 4 approaches
- Recommend one approach for Phase 12J
- Document security considerations
- Document token handling strategy
- Document error handling
- Document rate limit considerations

Expected Output:
- Design doc: docs/phase12/PHASE12J_NOTION_READONLY_SYNC_DESIGN.md
- Comparison table
- Recommended approach with justification
- Security assessment
- Implementation plan (if approved)

Note:
This is a design prompt. Do not implement sync without explicit human approval in a future phase.
